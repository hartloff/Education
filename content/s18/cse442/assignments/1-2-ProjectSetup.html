== Project Description

Provide a high-level outline of your project in the readme of your repository. This should give a clear idea of the vision for the project and a general idea of how that vision will become a reality. While writing this description keep you ideas separated into the following two categories/sections:

  1. MVP: As the name implies this is the minimal feature set needed to deliver a viable product. Think about the absolute minimal feature set that can still be useful to the end user and that is your MVP. This MVP feature set must be completed this semester so be careful what you put here. Though remember that you can change this after each project deadline so there's not too much harm in choosing a large MVP early.
  2. Add-on Features: These are additional features that can expand your project into it's full potential. The MVP combined with all add-on features should represent the full vision of what this project could become. Describe as many add-on features as necessary to describe your vision, but you must have at least 3 add-on features list in your readme. Note that you will not be required to implement any of these add-on features for the course so don't be afraid to think big.

Project Scope: As mentioned in lecture, the course staff reserve the right to deny any project proposal that is too trivial to justify a semester-long team project in a 400 level CSE course (ex. A desktop java app with the scope of a CSE116 project will be denied). In addition, the scope of each project will be taken into consideration when making decisions about the practical objectives. For example, a very simple project that is just ok might not earn a practical objective while an ambitious project that has some flaws may meet the objective.

== User Story

The readme for you project must also contain a user story for your MVP feature set. This story must be written in the perspective of the user and must be written in plain English (No technical language). This story describes the use case of the user and reflects what the user is trying to accomplish. The focus of this story is on the task that needs to be accomplished and not the solution for that task.

== Team Roles

Create a wiki for your project on GitHub and add a page named "Team Roles" that will define what each team member will be responsible to accomplish on your project. These roles must also include which team member will reviewing submitted code. You are free to choose your roles and who will review what code, but everything should be clearly defined and leave no ambiguity. When a new task needs to be completed it should be clear which one team member should be assigned to the task by referring to this page. These roles may be used during individual grading when necessary.

Since you will likely know more about your roles on the project as the semester progresses you should revisit these roles after each project submission and revise them if necessary.

== Dispute Resolution Protocol

You team must sign (with ink) and submit a paper copy of your team's dispute resolution protocol. This will be submitted during lecture on the due date for this submission.

Team members might argue over frameworks, languages, who's responsible for a specific task, or even whether to use tabs or spaces. Anything that is not clearly defined by the course or the project has the potential to become a dispute. You won't be able to predict every possible cause of a conflict later in the semester, but you must outline a protocol that you will follow if a dispute does occur. This can be something ridiculous, fun, serious, or whatever, but it must give a clear resolution. All members agree that this is the method that will be used to resolve disputes and will be bound to the results. This will help the team move forward when a deadlock is reached and neither side is willing to move off their positions.

Some examples:

  • Bring in a mediator (please define who will be the mediator)
  • Compete in a game (winner gets to make the decision)
  • Rock-paper-scissors or flip a coin (if you want random 50/50 decisions)

Whatever your team decides, it is recommended that you have fun with this section. You will be enforcing this protocol at times of high stress when you will likely need some light-hearted fun.
