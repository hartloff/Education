  1. Develop Beta Release: Develop the beta version of your project with a complete MVP feature set
  2. Deploy Beta Release: Deploy the beta version of your project for your beta testers
  3. Landing Page: Create a landing page for your project
  4. YouTube Video: Create and submit a YouTube video of your running software
== Practical Objective

Objective: Develop and release the beta version of your project with a complete MVP feature set and no major bugs. The release must be intuitive to use for a non-technical user.

Criteria:

  1. The software is easily accessed from the project's landing page
  2. The software is intuitive to use and provides the entire MVP feature set
  3. The software does not contain any major bugs
== Develop Beta Release

A non-technical user must be able to use the beta release of your software for the user story/stories of your project with all of your MVP features included and functional. For this release your software should be intuitive to use for a non-technical user.

Once the Beta version is complete, merge develop into master via a pull request and create a release from your master branch on GitHub as "v0.3.0"
== Deploy Beta Release

The beta version of your software must be available and accessible to a non-technical user. Your beta release can replace your alpha release and must meet the same requirements except it will be provided to your users through a landing page.

== Landing Page

Your beta release must be accompanied by a landing page for your project that includes instructions and details relevant to your beta testers. This site is separate from anything on the course website and you cannot assume visitors of this site know anything about your project. This implies that you cannot assume the users have seen any of your videos, though you can add any videos to your landing page if you want them to have that information.

This is the site you should/will share with anyone interested in the project and it can be thought of as a poster advertising your project. You can include any additional information you'd like to this page, but for the beta release you are required to have at least the following:

  • A self-contained description of your project
  • Instructions suitable for a non-technical user to access, run, and test your beta release
  • A clearly indicated link to your beta testing feedback form
  • A link to the issues page of your GitHub repository clearly marked as way to submit bug reports

You should also add any additional information that is specific to your project such as specific bugs to look for, which features to test, special testing credentials, etc. The more information you provide to your testers and the more specific questions you ask them the more likely you are to receive meaningful feedback.

*If your project is a web app the landing page can be the deployed app itself as long as it meets all the criteria of the landing page.

To submit, add a link to you landing page in the readme of your repository.

== YouTube Video

Upload a video demo of your beta release to YouTube that is shorter than 2 minutes. This video must give an accurate demo of the current state of your project and show your running software. This video will be directed towards a non-technical audience. You should not show any code or use any technical language in this video.


Submission

Add a link to your video in your project's readme before the deadline. You may remove any links to previous videos

== Testing

Although your testing at the team level will not be graded directly, you will be held accountable for any bugs or errors that appear in your released software. For your beta release there may be many minor bug reports (this is the purpose of beta testing), but you should not release your beta version with any major bugs or broken features. The bugs reported by beta testers should be such that they would have been very difficult to find in development.

It is highly recommended that your team agrees on a robust testing methodology so you can release your software with confidence. In addition, robust testing will make code reviews much simpler for the team. At the very least, it will save the team a lot of time since you can limit the number of times you'll need to go through the Hot Fix protocol.
